Nim Game
November - December 2023

Our project objective for the Nim game was to create a console-based Java implementation of the classic mathematical strategy game, Nim. The goal was to develop a program that allowed two players to play against each other and also incorporated algorithms for optimal gameplay/data analysis.

My role in this project was a programmer and tester. I was responsible for developing the core game logic, implementing the mathematical algorithms, and performing testing to ensure the game's functionality. My initial focus was on making the game as user friendly as possible for being a TUI-based piece of software, but I eventually found myself interested in the mathematical background behind "solutions" to the game. This led me to explore augmenting the program to conduct data analysis on solutions, which also led me to use Java design patterns to make the code neater with all its extra functionality. 

After exploring the algorithm behind the most optimal Nim solution (which roughly follows a pattern of 2^n - 1), I decided to train a Tensorflow neural network with a loss function that roughly mimicked the 2^n - 1 function in an attempt to see if I could get the neural network to perform matrix multiplication that somewhat followed the 2^n - 1 rule.

Making that neural network proved to be the hardest learning curve for this project. I have a lot of experience constructing models that work in a supervised manner with a training dataset, but I had never explored actively using a neural network that trained itself on a simple mathematical concept with really no training data. By the end, I got to the model to a ~30% win rate, but discovered that if I then reversed the decisions of the model's output I could get it up to a 60-70% win rate, which is about what the 2^n - 1 algorithm's win rate was in the first place.

Collaboration was a critical part of this project. Using Visual Studio Code's Live Share feature, my partner and I used pair programming, which allowed us to work on different parts of the codebase together. This approach allowed for real-time feedback and quick problem-solving, but required effective communication and coordination to program at the same time.

As mentioned before, the project also involved the use of design patterns such as interfaces, factories, and singletons to enhance code efficiency and maintainability. These patterns allowed for easier modification and extension of the gameâ€™s functionality, particularly in the creation of new algorithms as singleton classes.

This project taught me a lot about the importance of teamwork and planning in a large pair-programming project, but also about just how wide the field of computer science is - I took a simple game meant to teach us how to use Java objects in a practical manner and managed to explore design patterns, data analysis, terminal interactions, mathematical algorithms, and even neural networks/machine learning. It let me explore and appreciate every aspect of computer science, and is definitely the most fun little programming project I have ever worked on!

Exploratory Data Analysis
February 2024

Our projective objective for the PLTW 3.7.7 Data Lab project was to use a publicly available dataset, process/analyze it with Java, and present our conclusions/results from the data.

My role in this project was to handle data processing and analysis. I was responsible for managing the large datasets, performing data cleaning, and conducting statistical analysis to extract meaningful insights. The data source we used was a comprehensive dataset from Kaggle, which included information on all natural disasters from 1900 to 2021.

I began with data import and storage. Given the large dataset size, we used the Univocity library (we had to use Maven for package management) for CSV reading and batch processing. This allowed us to handle over 2 million data points by breaking the data into manageable chunks. We then imported this data into a local SQL database to for querying and manipulation.

To establish our database connection, we used JDBC (the basis of any Java SQL driver) for connecting to the SQL database. With the data now accessible in the database, we performed data retrieval by querying the database to extract relevant data subsets, focusing on analyzing disasters on a per-country basis over time in a stacked bar chart.

Overall, this project was short but taught me a lot about using SQL drivers to manage connections between Java and SQL databases, and also then using a library like JFreeCharts to graph the resulting data.
